# ADR-027: Ship a PRD template for offline / external-LLM drafting

**Status:** accepted
**Date:** 2026-04-26

## Context

[ADR-003](adr-003-prd-intake-model.md) established three PRD-intake
paths into the kit's planning flow: **no PRD** (use `idea-to-prd`
to generate one from a rough idea), **standard PRD** (a user-supplied
document in a conventional shape, normalised by `prd-normalizer`),
and **custom PRD** (any planning artefact, also normalised).

The "standard PRD" path is currently shape-implicit — the user is
told their input "should look conventional," but there is no
canonical shape to point at. In practice this leads to two failure
modes:

1. Users improvise an ad-hoc PRD shape that does not map cleanly
   onto `prd-normalizer`'s 11 canonical fields. The normaliser does
   more reshaping work than necessary, and the user is asked
   clarifying questions for fields the user could have filled in
   directly.
2. Users wanting to draft offline — for example, by handing a brief
   to an external LLM (Perplexity, ChatGPT, Claude.ai, Gemini, or
   any other), or by writing the PRD by hand in a Markdown editor
   without invoking the kit at all — have nothing concrete to work
   from. They have to read `prd-normalizer/SKILL.md` to discover
   the expected shape, then construct it themselves.

A canonical PRD template solves both: it gives users the shape
explicitly, makes offline drafting trivial, and means
`prd-normalizer` operating on a faithfully-filled template is
near-pass-through.

The template is **model-agnostic.** External LLMs are mentioned in
this ADR only as one example of an offline-drafting context. A user
filling the template by hand in their editor of choice is an
equally first-class use case. Nothing about the template depends on
which model (or human) does the drafting.

## Options considered

### Option A: Do not ship a template; rely on `prd-normalizer` to reshape any input

- Pros: zero new artefact to maintain; preserves the current "any
  shape works" message.
- Cons: users have to discover the canonical shape by reading the
  skill's documentation; offline drafters get no concrete starting
  point; `prd-normalizer` does avoidable reshaping work on every
  PRD that wasn't generated by `idea-to-prd`.

### Option B: Ship a PRD template that mirrors the 11 canonical fields

- Pros: gives the standard PRD path a concrete shape; offline
  drafting (with any external LLM, or by hand) becomes a fill-in
  exercise; `prd-normalizer` becomes a near-pass-through when the
  template is filled faithfully; sets clear expectations for what
  `[TBD]` is acceptable on which fields; the template can be
  handed verbatim to any external LLM as the prompt for "draft a
  PRD for my idea."
- Cons: one more template to maintain; needs to stay in sync with
  `prd-normalizer`'s canonical fields if those change.

### Option C: Ship a PRD template plus an "external LLM prompt" wrapper

- Pros: adds a ready-made instruction wrapper that prefixes the
  template with directives like "fill every field below; mark
  unknowns as [TBD]; do not invent content."
- Cons: couples the template to an "ask an LLM" framing and
  complicates the offline-by-hand use case; the template's own
  comment header can already do this work; over-engineered.

## Decision

Adopt **Option B**. Ship a PRD template at `templates/prd-template.md`
that mirrors `prd-normalizer`'s 11 canonical fields one-to-one. The
template:

- Lists every canonical field as a Markdown section with a
  placeholder body.
- Marks fields as **hard-required** (Product name, Problem, Primary
  user, at least one core capability, at least one user story) vs
  **soft** (everything else, where `[TBD]` is acceptable).
- Includes a leading HTML-comment header that explains how to use
  the template, which fields are hard-required, and where the
  filled file should land in a target project (`design/prd.md`).
- Is **model-agnostic**: the template itself contains no
  instructions to any specific LLM. The header notes the template
  can be filled "by the user, or by an external LLM" without
  naming any provider.

Reference the new template from:

- `templates/README.md` — index row pointing at the file and naming
  its consumer (`prd-normalizer`).
- `skills/prd-normalizer/SKILL.md` — under "Intake protocol —
  standard PRD path", note that a faithfully-filled
  `templates/prd-template.md` makes the normalisation step a
  pass-through.
- `skills/idea-to-prd/SKILL.md` — under the "When to use this skill"
  section, note that users with a PRD already drafted (by any means)
  should skip directly to `prd-normalizer` and may find
  `templates/prd-template.md` useful as a canonical shape.

The template's structure is canonical: changes to the 11-field
normalised PRD shape (in `prd-normalizer`'s SKILL.md) require a
matching change to this template, and vice versa. ADR-026's
versioning policy classifies a structural change to the template as
**MAJOR** (parser-relevant; `prd-normalizer` reads section headings
when consuming the standard-PRD path).

This ADR does not introduce a new PRD-intake path. The three paths
from ADR-003 — no PRD, standard PRD, custom PRD — remain. The
template makes the **standard PRD** path concrete; it is one
artefact along that path, not a new path.

## Consequences

- A new template ships at `templates/prd-template.md`. Filename and
  placement are inside `templates/`, alongside the MVP, README,
  CLAUDE.md, AI-summary, ADR, and PR templates already there.
- `templates/README.md` gains a row pointing at the file and naming
  its consumer.
- `prd-normalizer` and `idea-to-prd` SKILL.md files reference the
  template at the appropriate decision points.
- Offline drafting becomes a first-class flow: any user — with any
  external LLM, or none — can fill the template directly into
  `design/prd.md` and run `prd-normalizer` from there.
- `prd-normalizer` operating on a faithfully-filled template is
  near-pass-through; it confirms the two identity fields and writes
  the canonical file through. This shortens the planning loop.
- The template's structure now constrains future changes to
  `prd-normalizer`'s 11-field shape: changes propagate to the
  template (and trigger MAJOR per ADR-026 if they rename a
  parser-relevant heading).
- `notes/feature-ideas.md` entry **#16 (PRD template for external-LLM
  drafting)** moves from `idea` to `shipped`, linked to this ADR.
- The kit ships v3.1.0 (MINOR per ADR-026: new template, no
  breaking changes).
- This ADR is intentionally tool-agnostic. Future revisions could
  optionally add a separate guide doc on "drafting offline with an
  external LLM" with concrete prompting tips, but that guide would
  be advisory; nothing in the template, the skills, or the
  installer should depend on a specific external tool.
