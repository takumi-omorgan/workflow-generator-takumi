# Repository Structure

This document explains two things:

1. the layout of this **kit repository** (what you are looking at now), and
2. the layout of a **target project** after the kit has been installed into it.

It also says which artifacts stay in the kit and which get copied into a target
project. The split is governed by [ADR-001](../Design/adr/adr-001-project-local-installation-model.md)
— the workflow kit uses a **project-local installation model**. Each new project
gets its own fresh install; nothing is installed globally.

---

## Kit repository layout

```text
workflow-generator/
  README.md                  ← overview and quick start for the kit itself
  CLAUDE.md                  ← rules for Claude Code when working in this repo
  generic-project-workflow.md ← reference workflow the kit implements
  Claude Code Workflow Kit MVP Spec.md
  Claude Code Workflow Kit — Build-Out Plan.md
  Design/
    adr/                     ← accepted ADRs that govern the kit
      README.md
      adr-001-...md … adr-006-...md
  docs/                      ← kit documentation (install, github-setup, guides)
    README.md
    repo-structure.md        ← this file
  skills/                    ← source of the Claude Code skills the kit ships
    README.md
    <skill-name>/SKILL.md    ← one directory per skill (added in later issues)
  templates/                 ← starter templates copied into target projects
    README.md
  examples/                  ← worked examples for the three PRD intake paths
    README.md
  notes/                     ← working notes: issue drafts, per-issue prompts,
                               scratch setup files (kit-repo-only)
  .github/                   ← issue / PR templates for contributions to the kit
    ISSUE_TEMPLATE/
      feature-request.md
      docs-task.md
    pull_request_template.md
  .claude/                   ← Claude Code config for this repo
```

### What each top-level thing is for

| Path | Purpose | Copied into target projects? |
|---|---|---|
| `README.md` | Kit overview / quick start | No |
| `CLAUDE.md` | Claude Code rules for working **inside the kit repo** | No — a target project gets its **own** `CLAUDE.md` generated from `templates/claude-md-template.md` |
| `Design/adr/` | Accepted decisions about the kit | No — target projects get their **own** `Design/adr/` populated by `adr-writer` |
| `docs/` | Documentation about the kit | No |
| `skills/` | Source of each skill | **Yes** — copied to `.claude/skills/` in the target project |
| `templates/` | Starter templates | **Yes, selectively** — templates are copied or rendered by skills into the target repo |
| `examples/` | Reference examples | No |
| `notes/` | Working notes for building the kit itself | No |

## Target project layout (after installing the kit)

This is what a brand-new project looks like **after** the user has installed the
kit and run the initial planning skills.

```text
my-project/
  README.md                  ← generated by the workflow-docs skill
  CLAUDE.md                  ← rendered from templates/claude-md-template.md
  Design/
    adr/
      adr-001-...md          ← drafted by the adr-writer skill
      ...
    ai-summary.md            ← AI-readable summary of the project
  docs/                      ← optional: project-specific docs
  prompts/                   ← issue session briefs (one per GitHub issue)
    _template.md             ← blank prompt template (copy to start a new one)
    issue-<nnn>-<slug>.md    ← one filled prompt per GitHub issue worked on
  notes/                     ← freeform working notes (not tied to a session)
  .claude/
    skills/                  ← ALL the kit's skills, copied from skills/
      idea-to-prd/SKILL.md
      prd-normalizer/SKILL.md
      prd-to-mvp/SKILL.md
      adr-writer/SKILL.md
      issue-planner/SKILL.md
      workflow-docs/SKILL.md
      claude-issue-executor/SKILL.md
      pr-review-packager/SKILL.md
  .github/
    ISSUE_TEMPLATE/          ← from templates/ (added in Issue #9)
    pull_request_template.md
```

### What came from the kit vs. what the project generated

| Artifact in the target project | Source |
|---|---|
| `.claude/skills/*` | Copied from the kit's `skills/` |
| `.github/*` templates | Copied from the kit's `templates/` (via Issue #9) |
| `CLAUDE.md` | Rendered from `templates/claude-md-template.md` |
| `README.md` | Generated by `workflow-docs` skill |
| `Design/adr/*` | Drafted by `adr-writer` skill |
| `Design/ai-summary.md` | Generated from `templates/ai-summary-template.md` |
| `prompts/_template.md` | Copied from the kit (see ADR-008, Issue #12) |

Nothing in the target project points back at this kit repo. After install, the
target project is self-contained and can evolve independently.

## Installation flow (structural overview)

This is a structural sketch, not a full install guide — the full guide is
Issue #2. At a high level:

1. The user creates a new target project directory (or `gh repo create`).
2. They clone or download this kit repository to a scratch location.
3. They copy:
   - `skills/*` → `<target>/.claude/skills/`
   - selected templates from `templates/` → `<target>/templates/` or rendered
     directly into files like `<target>/CLAUDE.md`
4. They commit the result inside the target project.
5. From that point on, the kit's source is not required — the target project
   runs on its project-local copy of the skills.

Each new project repeats this process. There is no shared global install, no
symlink to a central skills directory, and no dependency on the kit being
present after install. That is what "project-local" means in ADR-001.

## Frequently-confused pairs

| Looks similar, but | Lives where |
|---|---|
| `CLAUDE.md` for the **kit** | `<kit>/CLAUDE.md` |
| `CLAUDE.md` for a **target project** | generated from `<kit>/templates/claude-md-template.md` |
| `Design/adr/` for the **kit** | `<kit>/Design/adr/` |
| `Design/adr/` for a **target project** | created inside the target project by `adr-writer` |
| `.claude/skills/` | **Only** exists inside a target project; the kit ships its skills under `skills/` (no leading `.`) |
