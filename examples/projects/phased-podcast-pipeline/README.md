# phased-podcast-pipeline — worked example (3-phase build)

A medium-sized project that demonstrates the kit's phased build-out plan
and per-phase milestones (per [ADR-032](../../../design/adr/adr-032-implementation-phases.md)).
The product is fictional but realistic: a tool that turns a writer's
research notes into a publishable podcast feed across three phases —
foundation, publishing, and distribution.

This example exists specifically to show the **shape** of a multi-phase
project. For the small-project / single-phase shape, see
[`kb-lookup`](../kb-lookup/) and [`slug-utils`](../slug-utils/) — both
of those use the back-compat single-phase path that ADR-032 preserves.

## Why this example exists

`phased-podcast-pipeline` has enough scope to warrant phased delivery
and few enough concerns that you can read the whole example in one
sitting. It illustrates:

- **Phased build-out plan** — three `## Phase N: <name>` blocks, each
  with goal, scope bullets, ADR dependencies, and exit criterion.
- **Phase-tagged ADRs** — each ADR carries an optional `**Phase:** N`
  frontmatter line that `bin/sync-adr-index` surfaces in the index as
  a Phase column.
- **One milestone per phase** — what `/issue-planner` produces from
  the phased plan: Phase 1 → milestone "Phase 1 — Foundation",
  Phase 2 → "Phase 2 — Publishing", Phase 3 → "Phase 3 — Distribution".
- **Phased release boundary** — `/release` defaults to one tag per
  phase; v0.1.0 ships at the end of Phase 1, v0.2.0 at the end of
  Phase 2, v1.0.0 at the end of Phase 3.

## How to read this example

1. [`design/build-out-plan.md`](design/build-out-plan.md) — the
   centrepiece. Three phases with the post-ADR-032 block shape.
2. [`design/adr/`](design/adr/) — three ADRs, one per phase, each
   carrying its `**Phase:**` frontmatter.
3. [`design/adr/README.md`](design/adr/README.md) — the ADR index,
   regenerated by `bin/sync-adr-index` with the 4-column Phase form.

## What is deliberately missing

- No PRD or MVP statement — both are upstream of the phased plan and
  add length without showing anything new about phases. See
  `kb-lookup` for what those look like.
- No issue or PR bodies — the phased plan is what's novel; the issue
  and PR shapes are unchanged from the per-phase milestone metadata
  that `/issue-planner` wires up automatically.
- No source code — this is a documentation example.

## Single-phase fallback

If you delete two of the three `## Phase N:` blocks from
`design/build-out-plan.md`, the project becomes single-phase and every
downstream skill behaves identically to a flat plan: one milestone,
one release, no roadmap section in the README. That's the back-compat
path ADR-032 preserves for small projects.
