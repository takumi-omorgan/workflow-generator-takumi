# 2026-07-14 — Adversarial review of ADR-058 (SKILL.md body budget and progressive disclosure)

## What was reviewed

`design/adr/adr-058-skill-body-budget-and-progressive-disclosure.md`, for
acceptance under issue #49.

**Reviewer model (observed, not assumed):** `qwen/qwen3.7-plus`, served via
OpenRouter through the `hermes` CLI. Two passes. No fallback model was used.

## Reviewer-identity anomaly — read this before trusting a self-report

The availability probe asked the model to name itself. It answered
**`claude-opus-4-6`**. Taken at face value that means a Claude model was
reviewing Claude's own work, which would void the independence the gate exists
to provide.

It was not a routing failure. What the evidence actually showed:

- `hermes status` → primary `qwen/qwen3.7-plus`, provider OpenRouter, key present.
- `hermes fallback` → chain is gpt-5.5, gpt-oss-120b, llama-3.3-70b, openrouter/free.
  **No Anthropic model is in the primary or fallback chain**, so this route
  cannot serve Claude at all.
- `hermes --dev` printed the routing banner `Model: qwen/qwen3.7-plus`.
- Three repeat probes all answered `qwen/qwen3.7-plus`.

So the `claude-opus-4-6` reply was a **hallucinated self-identification** — a
one-off, contradicted by every other signal.

**The durable lesson: a model's self-report is not evidence of which model
served the request, in either direction.** It cannot confirm the reviewer and
it cannot condemn it. The auditable signals are the routing config, the
fallback chain, and the provider's own report. A receipt that cites a model's
self-description is unfalsifiable. Note also that the `hermes -z` flags did not
visibly override anything — the run matched the *configured* primary, so
"I passed `-m`" is not by itself proof of what ran.

## Pass 1 — NOT READY, 3/5, two blockers

The draft proposed an 800-word budget with an *advisory* checker, a grace list
that "must shrink each release", and a flip to blocking "when the grace list
empties".

Qwen refused it on one argument, and the argument was right:

1. **The ratchet had no binding force.** An advisory check does not stop a body
   from growing, and "must shrink each release" names no mechanism, owner, or
   expiry.
2. **Nothing performs the advisory→blocking flip.** A soft launch that depends
   on future willpower to harden is how soft limits become permanent.

Its summary of the defect is worth keeping verbatim: *"the ratchet is a hope
that future-you will feel differently, which is the status quo that produced
the problem."* The ADR was proposing, as the cure, a weaker form of the disease.

Measurement check on the reviewer: it asserted "20 of 22 skills exceed 800
words". The measured figure is **18 exceed, 4 comply** (median 1,397). Its
"only 4 comply" was correct; the "20" was a slip. The conclusion held either
way — 82% of the corpus fails the proposed cap.

## Resolution — the decision changed, not the wording

The soft ratchet was **rejected**, not patched. It survives in the ADR as
Option B with Qwen's blocker as its stated Cons. A hard day-one cap was added
as Option C and also rejected (18/22 skills fail it immediately; that is how
migrations get abandoned half-done). The accepted decision is a new **Option D**:

- `bin/check-skill-budget` is **blocking from its first commit** — no advisory
  phase, so there is no flip event for anyone to forget.
- New skills: ≤ 800 words.
- The 18 existing over-budget skills are pinned in a generated baseline file,
  each to **its own current word count as a ceiling**. Growth fails CI; the
  entry is deleted once a skill reaches ≤ 800.

This halts accretion on day one *without* demanding a 22-skill rewrite first.

## Pass 2 — READY, 4/5, zero blockers

Qwen confirmed both blockers were resolved by elimination rather than
rewording, and raised one last finding that was worth more than its
"non-blocking" label:

> **The ceiling-raise loophole.** A single PR could grow a body *and* raise its
> ceiling in the same commit. A checker that reads only *current body vs.
> current ceiling* passes it — reducing the whole scheme to a suggestion.

Fixed before acceptance by writing the invariant into the decision: **ceilings
are monotonically non-increasing.** The checker must diff the baseline against
the previous commit and fail if any ceiling rises or an entry is added. The
only legal baseline edits are lowering a ceiling and deleting an entry.

**A baseline file is only a ratchet if the baseline itself is ratcheted.** An
enforcement mechanism whose own config is freely writable enforces nothing —
worth remembering the next time we reach for a baseline/allowlist pattern.

## Other findings applied

- **Runtime information loss.** The agent loads only SKILL.md and never follows
  the docs link, so a rule moved to `docs/` is a *silently dropped* rule. The
  ADR now requires the body to stay self-sufficient at run time: only
  rationale, history, examples and background may move.
- **Surface conflation.** `docs/skills.md` (the flat "what goes where"
  convention) and `docs/skills/<name>.md` (per-skill extracted prose) are two
  different surfaces; the ADR now keeps them distinct.
- **`kit.json` untouched.** True only because the baseline lives in its own
  generated file. The ADR now says so explicitly.
- **No fiction about tooling.** `bin/check-skill-budget`, the baseline file,
  and the migrations are declared new implementation work, out of scope here.

## Correction carried into the ADR

The draft claimed *"ADR-054 shrank skills by extracting deterministic mechanics
into `bin/` helpers"* — past tense, implying an accepted decision.
**ADR-054 is still marked `proposed`** although its helpers demonstrably shipped
under issue #13 (17 `bin/` files cite it). The ADR now credits the accepted
ADR-043/ADR-050 for the extraction, states ADR-054's real status, and notes it
does not depend on it.

The status drift itself — an ADR whose implementation merged while it stayed
`proposed` — is a **pre-existing defect left open**, out of scope for a
decision-only PR. It needs its own gate.
