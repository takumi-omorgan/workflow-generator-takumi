# HALT — ADR-059 (target-project kit lifecycle): false load-bearing premise

**Artifact:** `design/adr/adr-059-target-project-kit-lifecycle.md` (untracked draft — **not** accepted, **not** merged)
**Date:** 2026-07-14
**Outcome:** **HALTED before review.** Escalated to Oliver. No ADR-059 issue, branch, or PR was opened.
**Blast radius:** ADR-060 (ship-loop adoption tier) is halted too — it is downstream (its Option C calls `doctor`, an ADR-059 surface).

## What happened

ADR-059 was drafted on a premise that verification falsified. The overnight plan
flagged this premise as UNVERIFIED and pre-committed to halting if correcting it
reached the Decision. It does.

**ADR-059 Context claimed:**
> "The receipt layer (`bin/write-receipt`) **already records what the installer
> wrote and at which version**, which is exactly the provenance an upgrader needs."

**ADR-059 Decision built on it:**
> "**The receipt layer becomes the source of truth for what the kit owns in a
> target project**, and `upgrade` / `doctor` / `uninstall` are built on it."
> …mechanism: "compute a three-way plan **per receipt entry** — *replace*
> (kit-owned, **unmodified since install**), *skip + note*, *never touch*."

## The premise is false four ways

| # | Claim | Verified reality |
|---|---|---|
| 1 | The installer records what it wrote | It writes **no receipt**. The only `write-receipt` hit in `bin/install-workflow-kit` (line 410) **copies the script** into the target's `.claude/bin/`; it never invokes it. |
| 2 | Installer receipts exist | `docs/receipts.md`: *"shipping `write-receipt` into target projects via the installer [is a] **deferred follow-up** recorded in ADR-050."* |
| 3 | Receipts are durable provenance | They are **gitignored**. `docs/receipts.md`: *"local, per-run state, **not committed artefacts**"*; `templates/gitignore.target:22` = `.claude/receipts/`. **A fresh clone has zero receipts**, so `upgrade` would have nothing to classify. |
| 4 | Receipts can express file ownership | The schema (`schemas/receipt.v1.yaml`, fixed by **accepted** ADR-050) is keyed by **work-unit**, not file: required fields are `schema, version, skill, key, status, timestamp`, where `key` is *"an issue number, a PR number, a milestone slug, or a release tag."* There is **no path list, no per-file version, no hash**. It cannot express "these 47 files were installed at v5.0.1 with these hashes" — and "unmodified since install" is unanswerable without one. |

ADR-050 (accepted) is explicit about what receipts are *for*: **"Receipts are for
the actions you cannot simply re-run."** Cat-1 file-writing skills deliberately
do **not** write them, because their outputs are *"local files, already
idempotent."* A durable file-ownership ledger is the opposite kind of artifact.

## Why this is a HALT and not a wording fix

The plan permits revising a draft's Context and Options; it requires a HALT if
the correction reaches the **Decision**. It does. ADR-059 does not need a
better-worded receipt layer — it needs a **different artifact**: a *committed*
install manifest/lockfile in the target (file-keyed, hashed, version-stamped,
written by the installer, surviving `git clone`).

Swapping "the receipt layer" for "a new committed manifest" rewrites the Decision
*and* the Consequences (which claim the receipt layer is merely *"promoted from
install bookkeeping to a load-bearing contract"* — it cannot be promoted to a
role its accepted schema cannot represent). Worse, making receipts committed and
file-inventoried **contradicts accepted ADR-050**, which decided the opposite;
house rules require a **superseding ADR first**, and ADR-059 neither mentions nor
supersedes it.

## The option space (for Oliver — deliberately NOT decided here)

Stating options is not choosing one. This needs a human decision:

- **A — Redraft ADR-059 around a committed install manifest** (e.g.
  `.claude/kit-lock.json`: installed paths + kit version + hashes, committed).
  Note **ADR-061's manifest is the kit-side declaration** of what *should* be
  installed and who owns it; this would be the **target-side record** of what
  *was* written and whether it has since changed. They compose; neither replaces
  the other.
- **B — Supersede ADR-050 first**, redefining receipts as committed,
  file-inventoried provenance, then build ADR-059 on that. (Larger blast radius:
  ADR-050's receipts serve idempotency for mutating skills — a different job.)
- **C — Split ADR-059.** `doctor` and `uninstall` need far less provenance than
  `upgrade` does; a read-only `doctor` could ship against ADR-061's manifest
  alone. `upgrade` is the piece that truly needs the ownership ledger.

## Durable lesson

**"The X layer already does Y" is a premise, not a citation.** ADR-059 named a
real artifact (`bin/write-receipt`, `schemas/receipt.v1.yaml` — both exist, both
accepted) and inferred a capability from its existence. Existence was true;
*shape*, *durability* and *coverage* were all false, and the Decision rested on
all three. The check that catches this is cheap and mechanical: **for every "X
already does Y", open X and find Y.** Here, three greps — does the installer call
it, is it committed, does the schema have the field — falsified it before a line
of review was spent.
