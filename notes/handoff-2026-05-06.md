## Important context the next session should know

1. **The kit dogfoods itself.** This kit ships skills + templates that target projects use, AND uses those same skills internally for its own development. So /prepare-issue / /claude-issue-executor / /pr-review-packager are how *this* repo's PRs get made.
2. **The /pause skill isn't installed in this kit's `.claude/skills/`.** Most other kit skills are (via the `~/dotfiles/claude-config/bin/link-skills` script, per CLAUDE.md). The next session can run `link-skills` if /pause becomes useful, but the dogfooding playbook intentionally omits some skills.
3. **`Design/state.md` does not exist in this kit repo.** State-management skills (like /resume) silently no-op for state.md per ADR-035. This handoff file substitutes for state.md across sessions.
4. **`bin/check-plan` is now the canonical pre-write gate** for adr-writer and prepare-issue (both migrated in PR #75). The slash-command `/check-plan` is a wrapper around the same eval module. There's a known PROMPT-C2/C6 bug in the eval lib (item 6 above) that the next session may hit when prepping issues with no ADR — workaround is to use `- File: none — <reason>` shape inside the multi-line ADR block.
5. **Saved feedback memories that stayed load-bearing this session:**
   - Always follow the kit's formal workflow (ADR→issue→prompt→executor) regardless of change size.
   - Plan-mode rhythm: pause+flag at threshold → user enters plan mode → propose plan → user approves+exits → execute → re-flag at next threshold.
   - Kit install: user creates folder first, then installs into it.
6. **Auto-mode permission contract (ADR-041, §7 of workflow-guide).** Cat-3 operations (gh pr create, gh issue create, gh release create, milestone close) require explicit `yes` regardless of mode. Cat-2 (significant-task plan-mode in claude-issue-executor) requires written acknowledgement on bypass. Cat-1 is substitutable. Front-matter on every shipped skill declares its category.
